‚óŹ Both done. Here's the answer to your question and a summary of the change.

  1. Should Lens/projections differ between a settings-edited vs. 
  onboarding-created portfolio?

  No. The Lens engine and the projection are pure functions of three inputs only
   ‚ÄĒ (positions, store, settings). Nothing in the pipeline knows or records how
  the portfolio was created. generate_lens_full() ‚Üí build_lens_output(positions,
   store, settings), and run_projection() takes tickers/weights/value. There's
  no "provenance" field anywhere.

  So identical portfolio state ‚Üí identical brief, CTAs, caution score, and 
  projection graphs, regardless of origin. Concretely, "identical state" means:

  1. Same position records. Both onboarding and Settings ‚Üí Add use the same
  PositionDialog, which builds the same dict (ticker, shares, current_price, 
  equity, sector, name). Onboarding's launch() saves them verbatim;
  save_positions() doesn't normalize or stamp anything. And refresh_data()
  recomputes current_price/sector/equity for every position on each refresh ‚ÄĒ so
   after one refresh, an edited and a fresh portfolio with the same holdings are
   byte-identical.
  2. Same settings ‚ÄĒ chiefly risk_tier and the lens_signals thresholds.
  3. Same cached market data ‚ÄĒ prices, earnings dates, dividend dates.

  Determinism is built in: sentence selection is a SHA-256 hash of portfolio
  state, and the Monte Carlo uses a fixed seed (np.random.default_rng(seed=42)),
   so the fan shape is stable run-to-run.

  The only ways the "same" portfolio can produce different output:
  - Market-data timing ‚ÄĒ if the quote/earnings/dividend cache refreshes between
  two tests, equity = shares √ó current_price shifts, which moves projections and
   can tip a threshold. (TTLs: quote ‚Čą refresh interval, earnings/dividends 24h,
   daily history 60 min.) This is a when-you-tested effect, not a
  how-it-was-created effect.
  - History-less tickers ‚ÄĒ the MC fallback for a ticker with <20 days of history
   seeds an RNG with abs(hash(ticker)), and Python salts string hashing per
  process, so that one fallback path can vary across app restarts. Normal
  tickers with real history are unaffected.

  One bonus finding worth flagging (not a provenance difference ‚ÄĒ it affects all
   portfolios equally): the performance / unrealized-P&L analyzer reads
  pos.get('price') for current price and treats equity as cost basis. But
  positions store the live price under current_price (not price), and equity is
  continuously recomputed to current value ‚ÄĒ so there's no preserved entry
  price. Net effect: that analyzer reports ~0% unrealized P&L for everything,
  meaning P&L-driven CTAs/sentences are effectively dormant. It's consistent
  across both creation paths, so it doesn't answer your question either way ‚ÄĒ
  but if you want P&L to actually drive Lens output, we'd need to store an
  entry/cost-basis price separately at add time. Let me know if you want me to
  fix that.

  2. Sharpe widget tier rows

  In vector/widget_types/sharpe_ratio.py I trimmed the interpretation rows from
  <range> ‚ÄĒ <qualifier> ‚ÄĒ <explanation> to <range> ‚ÄĒ <qualifier>:

  - > 2.0 ‚ÄĒ Excellent
  - 1 ‚Äď 2 ‚ÄĒ Good
  - 0 ‚Äď 1 ‚ÄĒ Sub-optimal
  - < 0 ‚ÄĒ Poor

  The qualifier words stay (and still highlight on the active tier); only the
  trailing descriptions ("strong risk-adjusted return", etc.) are gone. Compiles
   clean.

‚óŹ Done. Here's the diagnosis and fix.

  Root cause (not a band-aid ‚ÄĒ the actual bug)

  The buy amounts collapsed because the dilution target equaled the flag 
  trigger. The chain:

  1. risk_profile.py merges the settings default
  lens_signals.stock_concentration_pct = 35 over concentration.moderate, so a
  stock is flagged as over-concentrated when its weight is > 35%.
  2. cta_engine.py Priority 6 then computed how much to buy elsewhere to dilute
  it ‚ÄĒ using that same 35% as the target: v_total_new = v_stock / 0.35.
  3. Your MSFT sits at ~35.4% ‚ÄĒ just over the trigger ‚ÄĒ so total_dollars = 
  v_stock/0.35 ‚ąí total_equity ‚Čą $210, split across 3 sectors ‚Üí $70 each.

  In other words, the engine flagged the position for being above 35% and then
  asked "how much to buy to bring it back to 35%?" ‚ÄĒ which is ‚Čą nothing. The
  math confirms it exactly: $70 √ó 3 = $210, and total_dollars = E¬∑(w‚ąí0.35)/0.35 
  = $210 solves to w ‚Čą 35.4%. Priority 7 (sector concentration) had the
  identical collision (target = 0.50 against a 50% sector trigger).

  The fix

  The dilution target now sits below the trigger via a single factor
  (_CONCENTRATION_DILUTION_FACTOR = 0.75), applied to both Priority 6 and 7. So
  a 35% trigger dilutes toward ~26%; a 50% sector trigger toward ~37.5%. It
  scales with whatever threshold the user sets and never collapses to ~$0.
  Existing _cap_buy_amount() caps (25%/CTA, 50%/group) still bound the upper
  end, so functionality is preserved.
